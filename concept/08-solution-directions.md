# Solution Directions (Exploratory)

No commitment to any single path. All directions attach to the **trip lifecycle** and **initialization tiers**.

## 1. Trip token system (guardian-initiated)

Parent gets short code → driver confirms → optional child verbal confirm.

- **Pros:** No child phone; works on SMS.
- **Cons:** Driver can ignore; needs adoption.

## 2. Visible vehicle identity (windshield QR / barcode)

Scan → lookup active registered trip + permit status.

- **Pros:** Fast boarding verification.
- **Cons:** Static scan ≠ live safety; spoofing if not tied to active trip.

## 3. Route integrity / deviation alerts

Expected A→B registered; geofence deviation triggers guardian + authority alert.

- **Pros:** Addresses kidnapping/detour scenario.
- **Cons:** Connectivity; false positives; driver privacy.

## 4. Driver protection module

Risk-scored zones; destination lock before trip; panic for drivers; dashcam incentives.

- **Pros:** Addresses robbery-via-detour.
- **Cons:** Tracking resistance; dispute process needed.

## 5. Safety layer vs new platform

| Approach | Description |
|----------|-------------|
| **Layer** | API/SDK for existing operators, schools, cooperatives |
| **Platform** | Full regulated marketplace — heavier lift |

## 6. Community monitor layer (primary social wedge)

Scroll-feed or forum shell where sanitized trips receive **remote witness quorum**. Channel *macco* energy into structured confirms and escalations.

- **Pros:** Matches T&T online culture; beta labor pool; restores visible trust.
- **Cons:** Vigilantism risk; unpaid labor ethics; abuse at scale.

## 7. Gamified credentials

Guardian Angel Points / True Trini Points → ranks → shareable badges → remote-work portfolio signal.

## MVP choice

Direction **6 + 7** via scroll-feed MVP; directions **1–4** deferred. See [`../mvp/README.md`](../mvp/README.md).
